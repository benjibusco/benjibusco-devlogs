## Freelance Workspace Backend Architecture

### Projects as the Root Module

Started the first backend-backed Freelance Workspace module with Projects.

Projects were chosen first because they are the workspace root record. Future modules can reference this structure through `projectId`, including:

```txt
Tasks
Client Hub
Assets metadata
Future uploads/downloads
```

The goal was to establish a stable backend pattern before expanding the rest of the workspace.

### Feature-First Module Structure

Reviewed the staging API structure before implementation and confirmed the API fits a feature-first architecture better than a global layer-first architecture.

Created the Freelance Projects module under:

```txt
app/modules/freelance/projects/
```

With the structure:

```txt
projects.routes.js
projects.controller.js
projects.repository.js
projects.validator.js
projects.constants.js
projects.mapper.js
```

This keeps Projects backend logic grouped together and makes future Freelance modules easier to duplicate, adapt, and maintain.

## Freelance Projects Data Model

### Existing Frontend Model Preservation

Reviewed the first backend implementation plan and rejected a schema that drifted away from the existing Projects UI.

Discarded fields such as:

```txt
title
slug
priority
startDate
dueDate
completedAt
```

Kept the current project metadata model:

```js
{
  (id, clientName, projectName, status, nextAction, notes, createdAt, updatedAt, archivedAt, deletedAt);
}
```

Preserved the existing project status flow:

```txt
Lead
Discovery
Proposal Sent
Awaiting Approval
In Progress
Review
Completed
Archived
```

This keeps the backend aligned with the frontend instead of forcing the UI into a new schema.

## Backend CRUD Implementation

### Protected API Route Mounting

Mounted the Freelance Projects routes in `app/server.js` behind the existing auth middleware:

```js
const freelanceProjectsRoutes = require("./modules/freelance/projects/projects.routes");

app.use("/freelance/projects", requireAuth, freelanceProjectsRoutes);
```

This keeps all Freelance Projects API routes private and aligned with the existing authenticated Lab tooling model.

### Module Responsibilities

`projects.constants.js` stores allowed statuses, restore statuses, and the default restore status. Restore statuses intentionally exclude `Archived`.

`projects.mapper.js` isolates database-to-frontend field conversion:

```txt
client_name → clientName
project_name → projectName
next_action → nextAction
archived_at → archivedAt
deleted_at → deletedAt
```

`projects.validator.js` validates API input, including:

```txt
valid UUIDs
required create fields
valid statuses
unknown fields
blocked timestamp fields
blocked deleted fields
restore status rules
empty update bodies
```

`projects.repository.js` contains SQL-only data access for:

```txt
listProjects
findProjectById
createProject
updateProject
archiveProject
restoreProject
softDeleteProject
```

Delete behavior was implemented as soft delete only:

```txt
deleted_at = now()
```

`projects.controller.js` handles validation, repository calls, success responses, not-found responses, validation errors, and server errors.

Standardized success responses around:

```js
{
  success: true,
  data: ...
}
```

Standardized general errors around:

```js
{
  success: false,
  error: "Message."
}
```

Standardized validation errors around:

```js
{
  success: false,
  error: "Validation failed.",
  details: errors
}
```

### CRUD Routes

Created `projects.routes.js` with the full Projects endpoint set:

```txt
GET    /freelance/projects
GET    /freelance/projects/:id
POST   /freelance/projects
PATCH  /freelance/projects/:id
PATCH  /freelance/projects/:id/archive
PATCH  /freelance/projects/:id/restore
DELETE /freelance/projects/:id
```

The route structure supports normal CRUD behavior plus explicit archive, restore, and soft-delete workflows.
