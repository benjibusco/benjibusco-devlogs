### Backend-Ready Assets Architecture

Designed Assets with a future backend migration in mind.

Core architecture decisions:

- async from day one
- repository/storage adapter split
- backend-shaped data model
- no real uploads yet
- URLs/paths only
- soft archive/delete support
- project-linked through `projectId`

The UI calls a shared async data interface instead of directly accessing localStorage:

```js
await FreelanceAssetsData.listAssets();
await FreelanceAssetsData.getAsset(assetId);
await FreelanceAssetsData.createAsset(assetData);
await FreelanceAssetsData.updateAsset(assetId, updates);
await FreelanceAssetsData.archiveAsset(assetId);
await FreelanceAssetsData.deleteAsset(assetId);
```

This keeps the frontend ready to swap from localStorage to an API/database later.

### Assets File Structure

Planned the Assets file structure:

```text
freelance/

        ├── assets-data.js
        ├── assets-local-storage.js
        ├── assets-api.js
        ├── assets.js
        ├── new-asset.js
        └── edit-asset.js
```

### Assets File Responsibilities

Defined each JavaScript file’s role:

- `assets-config.js` — constants, storage key, storage mode, categories, types, sources, statuses
- `assets-local-storage.js` — async localStorage repository
- `assets-api.js` — future API repository shell
- `assets-data.js` — public data adapter used by the UI
- `assets.js` — main dashboard rendering
- `new-asset.js` — create asset form logic
- `edit-asset.js` — edit/archive asset form logic

The storage layer is isolated so the UI can eventually move from:

```text
assets.js
→ FreelanceAssetsData
→ localStorage repository
→ localStorage
```

to:

```text
assets.js
→ FreelanceAssetsData
→ API repository
→ Express route
→ PostgreSQL
```

without rebuilding the frontend.

### Assets Data Model

Planned a backend-shaped asset record:

```js
{
  id: "asset-123",
  projectId: "project-456",
  title: "NazBro homepage screenshot",
  category: "Screenshots",
  type: "Image",
  source: "Local Path",
  url: "/assets/freelance/nazbro-homepage.png",
  status: "Available",
  notes: "Current homepage before redesign.",
  createdAt: "2026-06-23T18:00:00.000Z",
  updatedAt: "2026-06-23T18:00:00.000Z",
  archivedAt: null,
  deletedAt: null
}
```

This model can later map cleanly to a `freelance_assets` database table.

### Soft Archive and Delete Planning

Included `archivedAt` and `deletedAt` from the beginning.

This supports safer workspace behavior and better future backend compatibility.

Default asset lists hide soft-deleted assets while archived assets can still be shown, filtered, or restored later.
