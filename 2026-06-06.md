## Deployment Observability

### Deployment Metadata + Timestamp Refactor

Refactored deployment metadata handling to properly separate:

- deployment execution timestamps
- cron/status verification timestamps
- deployment state persistence

Identified a long-standing architectural issue where deployment JSON only tracked:

```json id="h4zwde"
updatedAt
```

which represented the latest cron/status write rather than the latest successful deployment event.

This caused deployment cards and deployment summaries to display unstable or unavailable deployment timestamps.

### Durable Deployment Timestamp Architecture

Implemented a durable deployment timestamp model inside the reusable deployment framework.

Deployment metadata now distinguishes between:

| Field            | Purpose                                   |
| ---------------- | ----------------------------------------- |
| `updatedAt`      | latest deployment JSON write / cron check |
| `lastCheckedAt`  | latest cron verification cycle            |
| `lastDeployedAt` | latest successful deployment event        |

This stabilized deployment history tracking across the entire deployment dashboard system.

### Deployment Timestamp Persistence Logic

Updated `deploy-lib.sh` to:

- preserve `lastDeployedAt` during:
  - `up-to-date`
  - failed deployments
  - non-deployment cron checks

- update `lastDeployedAt` only during:
  - successful deployments

- maintain merge-friendly deployment JSON structure

- preserve environment-aware deployment behavior

### Deployment JSON Stability Improvements

Refactored deployment JSON writing behavior to preserve deployment history across recurring cron checks.

Added:

```bash id="lbny05"
previous_last_deployed_at()
```

to preserve historical deployment timestamps during:

```text id="ow8goz"
up-to-date
```

checks.

Added:

```bash id="0kz5mo"
commit_timestamp()
```

fallback logic so deployment metadata can reconstruct missing deployment timestamps from repository commit history when required.

This permanently resolved:

- deployment timestamps resetting to empty
- deployment history loss during cron refreshes
- unstable deployment summary timestamps

### Reflog-Based Deployment Timestamp Backfill

Implemented automatic deployment timestamp backfilling using:

```bash id="a3o7g9"
git reflog
```

This solved the issue where repositories already contained deployed commits before the new timestamp architecture existed.

Backfill order now follows:

1. existing `lastDeployedAt`
2. legacy deployment timestamp fields
3. Git reflog deployment history
4. current successful deployment event

This allowed deployment cards to recover historical deployment timestamps without manually rebuilding deployment history.

### Environment-Isolated Deployment Locks

Refactored deployment lock architecture to prevent staging and production APIs from blocking each other.

Previous architecture used:

```text id="zcfcqt"
/tmp/${TARGET}.lock
```

which caused deployment conflicts between staging and production monitoring systems.

Deployment locks now dynamically include API environment context:

```text id="m8olr2"
/tmp/api-dev-lab-production.lock
/tmp/api-production-lab-production.lock
```

This allows both API environments to independently monitor identical deployment targets simultaneously without interference.
