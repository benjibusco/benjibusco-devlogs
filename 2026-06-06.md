## Deployment Metadata + Timestamp Refactor

Refactored deployment metadata handling to properly separate:

- deployment execution timestamps
- cron/status check timestamps
- deployment state persistence

Identified a long-standing architectural issue where deployment JSON only tracked:

```json
updatedAt
```

which represented the latest cron/status write rather than the most recent successful deployment event.

This caused deployment cards and deployment summaries to incorrectly display unstable or unavailable deployment timestamps.

### New Deployment Timestamp Architecture

Implemented a durable deployment timestamp model inside the reusable deployment system.

The deployment metadata architecture now distinguishes between:

| Field            | Purpose                                   |
| ---------------- | ----------------------------------------- |
| `updatedAt`      | latest deployment JSON write / cron check |
| `lastCheckedAt`  | latest cron verification cycle            |
| `lastDeployedAt` | latest successful deployment event        |

This stabilized deployment history tracking across the entire deployment dashboard system.

### New Deployment Timestamp Logic

Updated `deploy-lib.sh` to:

- preserve `lastDeployedAt` across:
  - `up-to-date`
  - `failed`
  - non-deployment cron checks

- update `lastDeployedAt` only on:
  - successful deployments

- maintain merge-friendly deployment JSON structure

- preserve environment-aware deployment behavior

### Added Previous Deployment Preservation

Added:

```bash

```

to preserve historical deployment timestamps during:

```text

```

checks.

### Added Commit Timestamp Fallback

Added:

```bash
commit_timestamp()
```

fallback logic so deployment metadata automatically reconstructs missing deployment timestamps from deployed repository commit history when needed.

This permanently resolved:

- deployment timestamps resetting to empty
- deployment history being lost during cron refreshes
- unstable deployment summary timestamps

## Reflog-Based Deployment Timestamp Backfill

Implemented automatic deployment timestamp backfilling using:

```bash
git reflog
```

This solved the problem where repositories had already deployed commits before the new deployment timestamp architecture existed.

Backfill order now follows:

1. existing `lastDeployedAt`
2. legacy deployment timestamp fields
3. git reflog deployment history
4. current successful deployment event

This allowed deployment cards to recover historical deployment timestamps without rebuilding deployment history manually.
